## 1. Engineering Concerns

### Critical (must resolve before building)

**E1 — Confidence State machine has no transition rules**
The spec defines 7 confidence states but not what events trigger each transition.
Without explicit rules, two engineers implement them differently and Business
Confidence becomes inconsistent. Resolution: define transitions in spec before
implementation. See state machine diagram below.

**E2 — SalesRecord missing stock_lot_id (schema flaw)**
`SalesRecord` links to `product_id` but not to `stock_lot_id`. With multiple
StockLots per product (different batches, confidence states), you cannot determine
which lot was sold. This breaks FIFO, per-lot dead stock detection, and partner
accountability. **Add nullable `stock_lot_id` FK to SalesRecord now.**

**E3 — Business Confidence formula undefined**
The spec lists 9 factors (verified inventory, sales, missing prices, etc.) but not
the formula. Different approaches (weighted average vs. penalty-based) produce very
different scores. The breakdown visible to users must be legible: not "missing_prices:
12" but "missing prices on your top-selling products." Define the formula before
building the confidence calculator.

**E4 — Photo draft lifecycle has no promotion trigger**
A `draft_photo` StockLot has no quantity, price, or SKU. The spec captures it and
creates Action Queue tasks. But: what triggers promotion from draft to real inventory?
Who resolves it? Without a defined lifecycle, the Action Queue fills with unresolvable
drafts and confidence degrades permanently for heavy photo-flow users.

**E5 — Concurrent edit: StockLot.quantity has no optimistic locking**
Mutable canonical quantity with two family members editing simultaneously = silent
data loss. The audit log preserves history but doesn't prevent the overwrite. Add
optimistic locking (version column or `updated_at` check) before shipping multi-user.

### High (should resolve before Phase 2)

**E6 — Import pipeline: duplicate detection strategy undefined**
The spec mentions "duplicate checks" but no deduplication key. If a brand imports
200 rows, then re-imports with 50 additions, what's the match key — SKU? Name? Both?
Neither is reliable with messy data. Define the merge strategy before building import.

**E7 — Mobile web camera reliability**
The photo-first flow is the core differentiator. Photos come from phones. getUserMedia
(web camera API) is inconsistent across Android browsers and unreliable for multi-photo
batch capture on mobile web. A responsive web app is not the same as a mobile-native
camera experience. Flag this as primary UX risk for the photo capture flow.

**E8 — S3 presigned upload needs client-side validation**
Direct upload to S3/R2 bypasses server validation. No file size limit, no file type
check, no virus scan. Users will upload WhatsApp-forwarded images (compressed, renamed),
videos mistaken for images, and 20MB phone RAW photos. Define validation rules before
building the upload flow.

### Medium (address in phased implementation)

**E9 — Import pipeline job queue not scoped**
Async import requires a job queue. The implementation tech (Redis + Sidekiq,
BullMQ, pg-boss, etc.) affects deployment complexity and cost. Define this before
starting Phase 2.

**E10 — Business Confidence dual role**
The score serves two audiences with different needs: (1) the owner wants a gut-check
("am I organized?"), (2) the system uses it to prioritize Action Queue items. These
have different update frequencies and tolerance for staleness. Keep them as one cached
value for MVP but design the driver breakdown to serve both audiences.

## 4. Confidence State Machine

```
                    ┌─────────────────────────────────────────────┐
                    │           CONFIDENCE STATE TRANSITIONS        │
                    └─────────────────────────────────────────────┘

  TRIGGER                          STATE TRANSITION

  photo captured, no other info    →  photo_only
  photo + name added               →  draft_photo
  photo + name + qty added         →  draft_photo  (stays; needs price to advance)

  CSV/Excel import                 →  imported_unverified
  ERP export import                →  imported_unverified
  manual entry (any fields)        →  manually_entered

  physical count verified          →  count_verified
  sales recorded against lot       →  sales_reconciled (from count_verified only)

  two sources disagree on qty      →  conflict_detected (from any state)
  conflict resolved by user        →  returns to pre-conflict state or count_verified

  ┌─────────────────────────────────────────────────────────────┐
  │  photo_only ─────────────────────────────────────────────►  │
  │                                                             │
  │  draft_photo ────────────────────────────────────────────►  │  conflict_detected
  │                                  ▲                          │  ◄──────────────────
  │  imported_unverified ────────────┤  (any → conflict         │
  │                                  │   on disagreement)       │  ────────────────►
  │  manually_entered ───────────────┤                          │  (resolve returns to
  │                                  │                          │   pre-conflict state)
  │  count_verified ─────────────────┤                          │
  │       │                          │                          └─────────────────────
  │       ▼                          │
  │  sales_reconciled ───────────────┘
  └────────────────────────────────────────────────────────────
```

**Business Confidence weight by state:**

| State | Confidence Weight |
|-------|------------------|
| count_verified | 1.0 |
| sales_reconciled | 0.95 |
| manually_entered | 0.7 |
| imported_unverified | 0.5 |
| draft_photo | 0.2 |
| photo_only | 0.1 |
| conflict_detected | 0.0 (plus penalty) |

**Formula (recommended):**
```
Base score = weighted_average(confidence_weight × quantity_certainty_factor)
Penalty: -3% per product missing price
Penalty: -2% per 10% of inventory in conflict_detected state
Bonus: +2% per week streak of zero new conflicts

quantity_certainty_factor:
  exact = 1.0
  approximate = 0.7
  unknown = 0.0 (excluded from weighted average denominator)
```

## 7. Architecture Decisions

### Database: PostgreSQL (recommended)

**Why PostgreSQL:**
- Complex joins between Product, StockLot, InventoryEvent, SalesRecord are essential
- Sales intelligence queries are aggregation-heavy (GROUP BY product, size, date range)
- Postgres JSON columns handle `driver_breakdown` and `payload` without schema lock-in
- Optimistic locking via version column is trivial in Postgres
- pg-boss or similar gives you a Postgres-backed job queue for import jobs (one less
  infrastructure dependency vs. Redis)

**Do not use MongoDB.** The relational integrity between these models is load-bearing.

### Required Infrastructure

```
Postgres (primary DB)
S3 or Cloudflare R2 (photo storage)
Job queue: pg-boss (Postgres-backed) for MVP or Redis + BullMQ
Cache: Redis or in-process LRU for confidence score cache
CDN: For photo delivery (S3 signed URLs or R2 public with custom domain)
```

**pg-boss (Postgres-backed job queue) is recommended for MVP** — eliminates Redis as a
dependency, uses the same Postgres instance, and handles import jobs reliably. Switch to
Redis + BullMQ if you need sub-second job pickup latency (not required for imports).

### Indexing Priorities

```sql
-- Covers most inventory visibility queries
CREATE INDEX ON stock_lots(brand_id, inventory_status, confidence_state);

-- Covers confidence calculation aggregations
CREATE INDEX ON stock_lots(brand_id, confidence_state);
CREATE INDEX ON stock_lots(brand_id, product_id);

-- Covers audit log queries
CREATE INDEX ON inventory_events(stock_lot_id, event_type, created_at);
CREATE INDEX ON inventory_events(brand_id, created_at DESC);  -- business feed

-- Covers sales intelligence
CREATE INDEX ON sales_records(brand_id, product_id, date);
CREATE INDEX ON sales_records(brand_id, date DESC);

-- Covers product search
CREATE INDEX ON products(brand_id, sku);
CREATE INDEX ON products(brand_id, category);
```

## 9. Failure Modes

| Code Path | Realistic Production Failure | Test? | Error Handling? | User Impact |
|-----------|------------------------------|-------|-----------------|-------------|
| Import async job | Job queue goes down mid-import | No | No | Import silently stuck in "processing" forever |
| Confidence BG job | Job fails after a verification | No | No | Score never updates; user verifies 10 items, score stays same |
| S3 presigned upload | Network drops after URL generated | No | No | URL expires; photo silently not uploaded; draft created with missing photo |
| Concurrent quantity edit | Two devices write simultaneously | No | No | Silent data loss — last write wins |
| StockLot/event drift | Bug causes quantity mismatch | No | No | Confidence score is wrong; no alert |
| Dead stock query timeout | >10K SKUs with no index | No | No | Sales Intelligence tab hangs or errors |

**Critical gaps (no test AND no error handling AND silent failure):**
1. Import job stuck — user has no visibility
2. Confidence background job failure — score becomes permanently stale
3. Presigned URL expiry without photo — draft created with no photo, no retry

## 12. Future Scalability Considerations

**Business Confidence at 50K+ SKUs:**
The background job approach scales well. At 50K SKUs per brand, the aggregation
query needs the indexes above. Add a read replica for analytics queries if needed.

**InventoryEvent as append-only audit log:**
Events will grow without bound. At 1M+ events per large brand, add partitioning by
`created_at` (Postgres range partitioning). For MVP, a simple `created_at DESC` index
on `(brand_id, created_at)` is sufficient.

**Sales Intelligence at scale:**
Best seller and dead stock queries are GROUP BY aggregations. They hit indexes well
for brands under 10K SKUs. At 50K+ SKUs, move to materialized views refreshed every
hour alongside the confidence BG job.

**Photo storage:**
R2 / S3 scales without ceiling. The only consideration is CDN cost for preview
thumbnails. Generate and store a 400px thumbnail at upload time to avoid serving
8MB originals in list views.

**Multi-tenant isolation:**
Every query should include `brand_id` as the first column in the WHERE clause.
All indexes above are brand-scoped. This ensures Postgres uses the index and never
cross-contaminates tenants.
